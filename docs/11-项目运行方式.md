# 11 · 项目运行方式

## 11.1 运行环境

| 项 | 说明 |
| --- | --- |
| 游戏平台 | [Screeps](https://screeps.com/)（编程 MMO） |
| 语言 | JavaScript（Screeps 运行时，Node.js 沙箱） |
| 入口 | `main.js` 导出 `module.exports.loop`，由游戏引擎每 tick 调用 |
| 构建步骤 | **无** — 无需 npm/webpack，文件即模块 |
| 依赖管理 | **无第三方依赖** — 仅使用 Screeps 全局对象（`Game`/`Memory`/`console`/`_`） |

## 11.2 部署方式

仓库根目录即 Screeps 脚本目录，两种方式：

### 方式一：本地同步（推荐）

本仓库已位于 Screeps 客户端脚本目录：

```
c:\Users\45221\AppData\Local\Screeps\scripts\screeps.com\default
```

Screeps 客户端会自动同步本目录所有 `.js` 文件到游戏服务器。修改文件后保存即可在下一 tick 生效。

### 方式二：手动上传

在 Screeps 官网或客户端的脚本编辑器（Scripts 标签）中，将本目录所有 `.js` 文件作为多模块上传，入口为 `main.js`。

### 使用 grunt/gulp 同步（可选）

若需从其他目录同步，可参考 Screeps 官方的 `grunt-screeps` 或 `gulp-screeps` 工具配置 `screeps.json`：

```json
{
  "email": "your@email.com",
  "password": "your-password",
  "branch": "default"
}
```

## 11.3 启动流程

部署后 bot 在下一个 tick 自动启动：

1. **首 tick**：`state._cacheReady` 为 false → 全量构建 Creep 缓存 + 初始化矿点坐标缓存。
2. **后续 tick**：进入「防御塔 → 清理死亡 creep → 检测短缺 → 角色调度 → 自动孵化」循环。
3. **自动达稳态**：当各角色数量达到 `config.roleTargets` 目标后，孵化器停止孵化，房间进入自维持运转。

### 冷启动场景

| 场景 | 行为 |
| --- | --- |
| 全新房间（无 creep） | `state.allNames.length <= 3` 触发紧急模式，以 harvester 为主重启 |
| 基地毁灭后重建 | 同上，紧急模式 + 房间变更检测清理失效状态 |
| 迁移到新房间 | `_purgeRoomState()` 清理失效任务与锁，重置缓存，下 tick 重新初始化 |

## 11.4 调试

### 日志开关

| 模块 | 开关位置 | 日志内容 |
| --- | --- | --- |
| body 部件 | `config.bodyConfig.debug = true` | 每次孵化的部件组合与成本 |
| collector | `role.collector.js` 中 `CURRENT_LOG_LEVEL` | 采集/投放/卡住/让位日志 |
| transporter | `role.transporter.js` 中 `CURRENT_LOG_LEVEL` | 取货/送货/锁管理/重置日志 |
| Tower | `manager.tower.js` 中 `DEBUG = true`（默认开） | 维修/跳过日志 |
| 任务调度 | `task.scheduler.js` 的 `_log` | 任务分配/完成/失败/让位日志 |
| Spawn | `manager.spawn.js` 始终打印 | 孵化成功/失败日志 |
| Main | `main.js` 始终打印 | 房间变更/状态清理日志 |

### 日志级别

collector 与 transporter 使用分级日志：

```js
var LOG_LEVEL = { DEBUG: 0, INFO: 1, WARN: 2, ERROR: 3 };
var CURRENT_LOG_LEVEL = LOG_LEVEL.INFO;  // 修改此值调整输出级别
```

### 控制台调试

在 Screeps 游戏控制台可直接执行：

```js
// 查看当前状态
console.log(JSON.stringify(require('state')))
// 查看任务统计
console.log(JSON.stringify(Memory.tasks.stats))
// 查看缓存计数
console.log(require('cache.creep').count('collector'))
// 手动触发紧急模式
require('state').emergencyMode = true
```

## 11.5 配置调优

修改 [`config.js`](../config.js) 调整：

| 字段 | 调优建议 |
| --- | --- |
| `roleTargets.collector` | 调大提升能量采集吞吐（注意矿点饱和，每 source 上限 3 个） |
| `roleTargets.transporter` | 调大提升搬运效率（注意 CPU 占用） |
| `roleTargets.upgrader`/`builder` | 调大加快升级/扩张（占用孵化名额） |
| `spawnEnergyThreshold` | 不应低于 200（一个最小 body 单元成本） |
| `emergencyCreepThreshold` | 调大更易触发紧急模式（更激进的重启） |
| `bodyConfig.debug` | 调试孵化时设为 true |

### 角色数量与矿点容量关系

每个 source 最多 `MAX_COLLECTORS_PER_SOURCE = 3` 个采集者。默认配置 `collector: 6` 适合 2 个 source 的房间。若房间 source 数量不同，按 `source数 × 3` 调整。

## 11.6 已知限制

| 限制 | 说明 |
| --- | --- |
| 单房间 | 当前架构面向单房间，多房间需扩展 `state` 与调度逻辑 |
| Spawn 单一 | 主循环取首个可用 spawn，多 spawn 需扩展 `manager.spawn` 队列 |
| 无战斗角色 | `body.config.js` 预留了战斗角色模板但未启用 |
| 房间名硬编码依赖 | 已改为动态发现，但部分日志仍假设单房间 |

## 11.7 文件修改后的生效

由于 Screeps 的模块缓存机制，**修改文件后下一 tick 自动生效**，无需重启。但需要注意：

- `Memory` 中的状态会保留，若修改了状态结构（如 `Memory.tasks` 字段），可能需要手动清理或依赖 `init()` 的兼容逻辑。
- `state.js` 是瞬态的，每 tick 重新计算，修改后立即生效。
