# 05 · 管理器层

管理器层负责横切关注点：孵化（`manager.spawn.js`）与防御塔（`manager.tower.js`）。

---

## 5.1 manager.spawn.js — 孵化管理器

[`manager.spawn.js`](../manager.spawn.js) 负责根据各角色数量缺口自动孵化 Creep，支持动态调整 body 大小与紧急模式。

### 依赖

```js
var config = require('config');
var creepCache = require('cache.creep');
var bodyConfig = require('body.config');
var state = require('state');
```

### `run(spawnName)` — 主入口

1. 取 `Game.spawns[spawnName]`，不存在则 return。
2. **孵化器忙则跳过**：`if (spawn.spawning) return`。
3. **紧急模式检测**：`this._checkEmergency()`。
4. 取 `state.emergencyMode ? config.emergencyRoleTargets : config.roleTargets` 作为目标配置。
5. **高能量等待**（非紧急模式）：`_canWaitForHighEnergy` 返回 true 且 `energy < capacity * 0.8` 则 return，等待更多能量孵化更高级 creep。
6. **按优先级队列检查缺口**（顺序：harvester → collector → transporter → upgrader → builder → repairer）：
   - `current = creepCache.count(role)`（O(1)）。
   - `current < need && energy >= config.spawnEnergyThreshold` 时：
     - 生成名字：`角色首字母大写 + Game.time`（如 `Collector123456`）。
     - `body = this.getBody(energy, role)`。
     - `spawn.spawnCreep(body, name, {memory:{role}})`。
     - 成功 → `creepCache.add(name)` 同步缓存，打印 `[Spawn] 孵化 ...`。
     - 失败 → 打印 `[Spawn] 孵化失败 ... 错误码`。
     - **一次只孵一个**（return）。

### `checkShortage()` — 短缺检测

遍历 `config.roleTargets`，任一角色 `creepCache.count(role) < targets[role]` 返回 `true`。结果写入 `state.creepShortage`，供 builder/upgrader 暂停使用。

### `_checkEmergency()` — 紧急模式检测

| 状态 | 进入条件 | 退出条件 |
| --- | --- | --- |
| 正常 → 紧急 | `state.allNames.length <= config.emergencyCreepThreshold`（默认 3） | — |
| 紧急 → 正常 | — | `creepCache.count('harvester') >= config.emergencyRoleTargets.harvester`（默认 3） |

- 进入紧急模式：`state.emergencyMode = true`，打印 `[Spawn] 进入紧急模式（creep 总数: N）`。
- 退出紧急模式：`state.emergencyMode = false`，打印 `[Spawn] 退出紧急模式，恢复正常配置`。

> 紧急模式使用 `emergencyRoleTargets`（以 harvester 为主，不依赖 Container/transporter），且**不等待高能量**，立即孵化。

### `_canWaitForHighEnergy(spawn, targets)` — 高能量等待判定

满足以下**全部**条件才允许等待高能量：

1. `collectorCount + transporterCount >= (targets.collector + targets.transporter) / 2`（基础劳动力充足）。
2. `transporterCount > 0`（必须有搬运者）。
3. 房间内 `STRUCTURE_CONTAINER` 有能量（`store[RESOURCE_ENERGY] > 0`）。

### `getBody(energy, role)` — body 生成

委托给 [`body.config.js`](07-身体部件配置.md) 的 `getBody`，详见第 7 章。

## 5.2 manager.tower.js — 防御塔管理器

[`manager.tower.js`](../manager.tower.js) 遍历房间内所有 Tower，按优先级执行动作。

### 依赖

```js
var state = require('state');
```

### 常量

| 常量 | 值 | 说明 |
| --- | --- | --- |
| `NORMAL_REPAIR_THRESHOLD` | `0.8` | 普通建筑血量低于满血 80% 才修 |
| `DEFENSE_HITS_TARGET` | `50000` | Wall/Rampart 维修血量上限 |
| `TOWER_MIN_ENERGY_RATIO` | `0.5` | Tower 能量低于 50% 时不维修，保留防御 |
| `DEBUG` | `true` | 调试日志开关 |

### `run()` — 主入口

调用 `_getTowers()` 获取塔列表，逐个执行 `_runTower(tower)`。

### `_getTowers()` — Tower 缓存

带缓存的 Tower 获取，缓存命中时用 `Game.getObjectById`（O(1) per tower）：

1. `state.towerIds.length === 0` → 需要全量扫描（检测新建的 Tower）。
2. 否则逐个 `Game.getObjectById` 验证，任一失效（被摧毁）→ 标记 `needRefresh`。
3. `needRefresh` 时清空 `state.towerIds`，遍历 `Game.structures` 收集所有 `STRUCTURE_TOWER` 并写入缓存。

### `_runTower(tower)` — 单塔逻辑

**优先级：** `攻击敌人 > 紧急维修普通建筑 > 维修防御建筑`

```
1. tower.pos.findClosestByRange(FIND_HOSTILE_CREEPS)
   ├─ 有敌人 → tower.attack(hostile) → return（无视能量）
   └─ 无敌人 → 继续

2. 计算 energyRatio = energy / capacity
   ├─ energyRatio < 0.5 → 跳过维修，打印日志 → return
   └─ 能量充足 → 继续

3. findClosestByRange(FIND_STRUCTURES, 血量 < 80% 满血 且 非 Wall/Rampart)
   ├─ 有 → tower.repair(damaged) → return（紧急保护功能性建筑）
   └─ 无 → 继续

4. findClosestByRange(FIND_STRUCTURES, Wall/Rampart 且 hits < 50000)
   ├─ 有 → tower.repair(defense) → return
   └─ 无 → 打印日志（无 Wall/Rampart 或均已达标）
```

### 日志示例

```
[Tower] 能量不足(120/1000)，跳过维修，等待补充
[Tower] 维修普通建筑: road 血量:800/5000
[Tower] 维修防御建筑: constructedWall 血量:12300/50000
[Tower] Wall/Rampart 血量均已达标(50000)，当前数量:12
```
