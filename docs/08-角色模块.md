# 08 · 角色模块

所有角色模块导出 `{ run(creep) }`，由 `main.js` 按 `creep.memory.role` 分派调用。

## 8.1 role.harvester.js — 采集+运输（兼容旧逻辑）

[`role.harvester.js`](../role.harvester.js) — 当前目标数=0，主力采集改为 collector + transporter 分工。

### 依赖

```js
var sourceCache = require('cache.sources');
```

### 状态机

```
harvesting (能量空) ↔ transporting (能量满)
```

- `creep.store[RESOURCE_ENERGY] == 0` → `harvesting = true`
- `creep.store.getFreeCapacity() == 0` → `harvesting = false`

### 行为

| 状态 | 行为 |
| --- | --- |
| `harvesting` | `sourceCache.harvestNearest(creep)`（带满位顺延） |
| `transporting` | 优先补 Spawn/Extension/Tower，其次 Container |

## 8.2 role.collector.js — 纯采集

[`role.collector.js`](../role.collector.js) — 纯采集，不负责长途运输。

### 依赖

```js
var taskScheduler = require('task.scheduler');
var sourceCache   = require('cache.sources');
var state         = require('state');
```

### 内部常量

| 常量 | 值 | 说明 |
| --- | --- | --- |
| `MAX_COLLECTORS_PER_SOURCE` | 3 | 每 source 最大采集者数 |
| `STUCK_THRESHOLD` | 50 | 卡住检测阈值（ticks 无移动） |
| `RETRY_BASE_INTERVAL` | 5 | 重试退避基础间隔 |
| `MAX_SOURCE_FAIL_COUNT` | 3 | 连续无法到达同一 source 的最大次数 |
| `CACHE_TTL` | 20 | Room.find 缓存有效期 |
| `DROP_MAX_DISTANCE` | 3 | 投放 Container 的最大距离 |
| `CURRENT_LOG_LEVEL` | `INFO` | 当前日志级别 |

### 状态机

```
idle → harvesting → dropping → idle
```

- 能量空 → `harvesting`
- 能量满 → `dropping`

### run(creep) 流程

1. **让位检查**：`taskScheduler.checkYield(creep)` → return true 则跳过本 tick。
2. **健康检查**：`_healthCheck(creep, logCtx)` → return true 则跳过。
3. **状态切换**：能量空 → harvesting；能量满 → dropping。
4. **分派**：harvesting → `_doHarvest`；dropping → `_doDrop`。

### 关键方法

| 方法 | 说明 |
| --- | --- |
| `_doHarvest(creep, logCtx)` | 采集逻辑：距离检查 → 移动 → harvest，处理失败计数与让位 |
| `_doDrop(creep, logCtx)` | 投放逻辑：找范围内 Container → transfer，满了/无则 drop |
| `_getAssignedSource(creep)` | 源分配：保持绑定 → 满载则换近矿 → 兜底最近未冷却矿 |
| `_countCollectorsAtSource(sourceId)` | 统计某 source 当前 collector 数（含 harvester 兼容），用 `state.byRole` 缓存 |
| `_healthCheck(creep, logCtx)` | 卡住检测，进入恢复模式（20 ticks 冷却），重置所有状态并释放任务锁 |
| `_updateHealth(creep)` | 记录进展，重置退避计数 |
| `_handleHarvestError(creep, errorCode, logCtx)` | 指数退避重试（`5 * 2^(retryCount-1)`，上限 100 ticks） |

### 核心特性

1. **源分配与并发控制**：每 source 最多 3 个采集者，按 Spawn 距离近→远优先占用。
2. **纯采集投放**：能量满后投放到 `DROP_MAX_DISTANCE=3` 范围内的 Container，无 Container 则 `drop` 在地上（drop mining）。
3. **指数退避重试**：harvest 失败时按 `5 * 2^(retryCount-1)` 退避，上限 100 ticks。
4. **健康检查与自恢复**：50 ticks 无移动进入 20 ticks 恢复模式，重置所有状态并释放任务锁。
5. **Room.find 缓存**：sources/containers 缓存 20 ticks。
6. **结构化日志**：`LOG_LEVEL` 分级（DEBUG/INFO/WARN/ERROR），当前 `INFO` 级。
7. **让位协调**：`run()` 入口调用 `taskScheduler.checkYield(creep)`。
8. **连续失败切换 source**：3 次无法到达同一 source 时，先 `requestYield`，无让位者则换 source 并冷却 100 ticks。

### `_doHarvest` 失败处理流程

```
未到达 source (creep.pos.inRangeTo(source, 1) = false):
├─ creep.moveTo(source, {reusePath:10})
│   ├─ ERR_NO_PATH
│   │   ├─ _sourceFailCounts[source.id]++
│   │   ├─ failCount >= 3
│   │   │   ├─ taskScheduler.requestYield(creep, source)
│   │   │   │   ├─ 成功 → 清空失败计数，return
│   │   │   │   └─ 失败 → assignedSourceId=null，标记冷却 100 ticks
│   │   └─ failCount < 3 → 打印日志
│   └─ 其他 → 重置失败计数
└─ _updateHealth(creep)

到达 source:
├─ harvest(source) == OK → _updateHealth，接近满仓切 dropping
├─ ERR_NOT_ENOUGH_RESOURCES → _updateHealth（正常再生中）
├─ ERR_NOT_OWNER → 换 source
└─ 其他 → _handleHarvestError（指数退避）
```

## 8.3 role.transporter.js — 搬运

[`role.transporter.js`](../role.transporter.js) — 全场搬运，从采集节点到消费节点。

### 依赖

```js
var taskScheduler = require('task.scheduler');
```

### 内部常量

| 常量 | 值 | 说明 |
| --- | --- | --- |
| `CACHE_TTL` | 5 | Room.find 缓存有效期（短 TTL 快速发现新掉落） |
| `DELIVERY_PRIORITY` | extension=1, spawn=2, tower=3, storage=4, container=5 | 送货优先级 |
| `SPAWN_LOW_ENERGY` | 200 | Spawn 低能量阈值 |
| `TOWER_LOW_ENERGY` | 500 | Tower 低能量阈值 |
| `MIN_PICKUP_AMOUNT` | 20 | 最小搬运资源阈值 |

### 状态机（两阶段）

```
empty 阶段: GET_TASK → MOVING_TO_PICKUP → PICKING_UP
carry 阶段: DELIVERING → MOVING_TO_DELIVER → (executeDelivery) → COMPLETE
```

- `creep.store[RESOURCE_ENERGY] === 0` 且 `phase !== 'empty'` → 切 empty，重置任务与锁，`say('🔄 取货')`。
- `creep.store.getFreeCapacity() === 0` 且 `phase !== 'carry'` → 切 carry，`say('📦 送货')`。

### 取货优先级（`_findBestPickup`）

1. **地面掉落资源**（`dropped.amount >= 50`）— 防止消失，加锁。
2. **Tombstone / Ruin**（`store[RESOURCE_ENERGY] >= 20`）— 回收死亡 creep 资源，加锁。
3. **Container**（`store[RESOURCE_ENERGY] >= 100`）— `>= 500` 不加锁，支持多搬运者；`< 500` 加锁。
4. **Storage**（`store[RESOURCE_ENERGY] >= 500`）— 加锁。

### 送货优先级（`_findBestDelivery`）

`DELIVERY_PRIORITY`：extension(1) > spawn(2) > tower(3) > storage(4) > container(5)

- **Spawn 能量 < 200** → 提升到优先级 0（最高）。
- **Tower 能量 < 500** → 提升到优先级 2.5。
- 同优先级内按剩余容量降序（最空的优先）。
- 候选列表排序后，找第一个未达锁定上限（3 个）的目标。
- 兜底：所有消费端满/锁定 → Storage。

### 关键特性

1. **资源锁管理**（`Memory._transporterLocks`）：取货锁单锁，送货锁支持最多 3 个搬运者同时锁定。锁有 30 ticks 超时自动释放。
2. **让位检查**：`run()` 入口 `taskScheduler.checkYield`。
3. **健康检查**：停滞 > 50 ticks 强制重置状态，释放所有锁。
4. **移动错误处理**：连续 5 次移动失败放弃目标；取货/送货错误 3 次放弃。
5. **抖动修复**：未满容量不切送货阶段，避免 Container↔Spawn 往返抖动；无送货目标时保持 carry 阶段不切 empty。
6. **站在阻挡结构上移开**：`_moveAwayFromTarget` 手动计算相邻格，避免 `creep.pos.getAdjacentPosition` 不存在的问题。
7. **Room.find 缓存**：`CACHE_TTL = 5` ticks（dropped/tombstones/structures）。

### 关键方法

| 方法 | 说明 |
| --- | --- |
| `_doPickup` / `_moveToPickup` / `_executePickup` | 取货三阶段 |
| `_doDelivery` / `_moveToDeliver` / `_executeDelivery` | 送货三阶段 |
| `_findBestPickup(creep)` | 按优先级找最优取货点 |
| `_findBestDelivery(creep)` | 按优先级找最优送货目标 |
| `_isPickupLocked` / `_lockPickup` / `_releasePickupLock` | 取货锁管理 |
| `_isDeliverLocked` / `_lockDeliver` / `_releaseDeliverLock` | 送货锁管理（多锁，上限 3） |
| `_releaseAllLocks(creepName)` | 释放 creep 持有的所有锁 |
| `_moveAwayFromTarget(creep, target)` | 从目标位置移开一格（8 方向逐个尝试） |
| `_healthCheck(creep, logCtx)` | 停滞检测与强制重置 |

### `_moveAwayFromTarget` 8 方向遍历

```js
var directions = [TOP, TOP_RIGHT, RIGHT, BOTTOM_RIGHT, BOTTOM, BOTTOM_LEFT, LEFT, TOP_LEFT];
// 逐个尝试，找到边界内、非墙、无其他 creep 的格子 → creep.move(dir) → return
// 全部被堵 → creep.move(随机 1-8)
```

## 8.4 role.upgrader.js — 升级

[`role.upgrader.js`](../role.upgrader.js) — 升级控制器。

### 依赖

```js
var state        = require('state');
var sourceCache  = require('cache.sources');
var taskScheduler = require('task.scheduler');
```

### 常量

| 常量 | 值 | 说明 |
| --- | --- | --- |
| `SPAWN_CLEAR_DISTANCE` | 5 | 离开 Spawn 的距离（避免挡搬运者） |

### 状态机

```
upgrading (能量满) ↔ 装能量 (能量空)
```

- `upgrading && store < 1` → `upgrading = false`，`say('🔄 装能量')`。
- `!upgrading && store 满` → `upgrading = true`，`say('⚡ 升级')`。

### 行为

| 状态 | 行为 |
| --- | --- |
| `upgrading` | 短缺时 `_moveAwayFromSpawn`；否则 `upgradeController`，`ERR_NOT_IN_RANGE` 时 moveTo；`ERR_NOT_ENOUGH_ENERGY` 切回采集 |
| 装能量 | 优先 Container（按距离排序）→ Spawn/Extension → `sourceCache.harvestNearest` 直接采矿 |

### `_moveAwayFromSpawn(creep)`

- 找首个 spawn，`getRangeTo(spawn) >= 5` 直接 return。
- 否则计算远离 spawn 方向的偏移（`dx/dy` 归一化 × 5），边界保护 `max(1, min(48))`，`moveTo`。

## 8.5 role.builder.js — 建造

[`role.builder.js`](../role.builder.js) — 建造工地。

### 依赖

```js
var state        = require('state');
var sourceCache  = require('cache.sources');
var taskScheduler = require('task.scheduler');
```

### 状态机

```
building (能量满) ↔ 采集 (能量 < 5)
```

- `building && store < 5` → `building = false`，`say('🔄 采集')`（BUILD_POWER=5，避免残余能量死锁）。
- `!building && store 满` → `building = true`，`say('🚧 建造')`。

### 行为

| 状态 | 行为 |
| --- | --- |
| `building` | 短缺时 `_moveAwayFromSpawn`；否则找 `FIND_CONSTRUCTION_SITES`，`build(targets[0])`，无工地时升级控制器 |
| 采集 | 优先 Container（按距离排序）→ Spawn/Extension → `sourceCache.harvestNearest` |

> 注：builder 采集逻辑与 upgrader 完全一致，遍历所有可用结构直到成功取能量。

## 8.6 role.repairer.js — 修理

[`role.repairer.js`](../role.repairer.js) — 修理受损建筑。

### 依赖

```js
var sourceCache   = require('cache.sources');
var taskScheduler = require('task.scheduler');
```

### 常量

| 常量 | 值 | 说明 |
| --- | --- | --- |
| `DEFENSE_HITS_MIN` | 10000 | 防御建筑紧急修理阈值 |
| `DEFENSE_HITS_TARGET` | 50000 | 防御建筑目标血量上限 |
| `ENABLE_WALL_REPAIR` | true | 是否维修 Wall/Rampart |

### 状态机

```
repairing (能量满) ↔ 采集 (能量空)
```

- `repairing && store < 1` → `repairing = false`，`say('🔄 采集')`。
- `!repairing && store 满` → `repairing = true`，`say('🔧 修理')`。

### `_findRepairTarget(creep)` — 修理优先级

1. **紧急修理**：血量 < `DEFENSE_HITS_MIN=10000` 的 Wall/Rampart。
2. **普通建筑**：按血量升序，找第一个可到达的。
3. **防御建筑**：血量 < `DEFENSE_HITS_TARGET=50000` 的 Wall/Rampart（按血量升序）。
4. **普通建筑兜底**：按血量升序，找第一个可到达的。
5. **无目标**：升级控制器。

> 注：用 `creep.pos.getRangeTo(target) > 0` 过滤不可到达的目标。

### 采集模式

与 upgrader/builder 一致：Container → Spawn/Extension → `sourceCache.harvestNearest`。
