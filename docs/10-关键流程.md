# 10 · 关键流程

## 10.1 每 tick 主循环

```
loop()
  │
  ├─ 1. 重置 state.sourceSlotCount = {}
  │
  ├─ 2. Spawn 动态发现
  │   └─ 遍历 Game.spawns 取首个可用 spawn → state.spawnName/spawnRoomName
  │       └─ 房间变更? → _purgeRoomState() + 重置缓存 + state._cacheReady=false
  │
  ├─ 3. 首 tick 初始化 (!state._cacheReady)
  │   ├─ 无 spawn → return（等下一 tick）
  │   ├─ creepCache.build()  全量构建 Creep 名字缓存
  │   └─ 遍历房间 source → state.sourceIds / state.sourceData
  │
  ├─ 4. taskScheduler.init()  确保 Memory.tasks 结构 + GC
  │
  ├─ 5. managerTower.run()  防御塔：攻击 > 维修普通 > 维修防御
  │
  ├─ 6. 清理死亡 Creep
  │   └─ 遍历 Memory.creeps → 不存在则 creepCache.remove(name) + delete Memory.creeps[name]
  │
  ├─ 7. taskScheduler.checkTimeouts()  超时回收
  ├─ 8. taskScheduler.checkDeadlocks() 死锁检测
  │
  ├─ 9. state.creepShortage = managerSpawn.checkShortage()
  │      state.spawningRole = null
  │
  ├─ 10. 角色调度
  │   └─ 遍历 state.allNames → switch(creep.memory.role) → role*.run(creep)
  │
  └─ 11. if (state.spawnName) managerSpawn.run(state.spawnName)
```

## 10.2 孵化决策流程

```
managerSpawn.run(spawnName)
  │
  ├─ spawn.spawning? → 跳过
  │
  ├─ _checkEmergency()  紧急模式检测
  │   ├─ allNames.length <= 3 → 进入紧急模式（emergencyRoleTargets）
  │   └─ harvester 达标 → 退出紧急模式（roleTargets）
  │
  ├─ targets = state.emergencyMode ? emergencyRoleTargets : roleTargets
  │
  ├─ 非紧急 && _canWaitForHighEnergy(spawn, targets)?
  │   └─ energy < capacity * 0.8 → return（等待高能量）
  │
  ├─ 按优先级队列遍历: harvester → collector → transporter → upgrader → builder → repairer
  │   └─ count(role) < need && energy >= 200?
  │       ├─ name = 首字母大写 + role + Game.time
  │       ├─ body = bodyConfig.getBody(energy, role)
  │       ├─ spawn.spawnCreep(body, name, {memory:{role}})
  │       └─ 成功 → creepCache.add(name) + 打印日志
  └─ 一次只孵一个 (return)
```

### 紧急模式触发与退出

| 状态转换 | 条件 | 配置切换 |
| --- | --- | --- |
| 正常 → 紧急 | `state.allNames.length <= emergencyCreepThreshold(3)` | `roleTargets` → `emergencyRoleTargets` |
| 紧急 → 正常 | `creepCache.count('harvester') >= emergencyRoleTargets.harvester(3)` | `emergencyRoleTargets` → `roleTargets` |

紧急模式特点：
- 以 harvester 为主（目标 3），不依赖 Container/transporter。
- **不等待高能量**，立即孵化。
- upgrader/builder/repairer 目标数均为 0。

## 10.3 Transporter 状态机

```
empty 阶段（取货）
   ┌──────────────────────────────────────────────────┐
   ▼                                                  │
GET_TASK ──→ _findBestPickup ──→ MOVING_TO_PICKUP ──→ PICKING_UP
   │                                              │
   │   (未满容量)                                  │
   └──────────────────────────────────────────────┘
   │   (满容量 → 切 carry)
   ▼
carry 阶段（送货）
   ┌──────────────────────────────────────────────────┐
   ▼                                                  │
DELIVERING ──→ _findBestDelivery ──→ MOVING_TO_DELIVER ──→ _executeDelivery
   │                                              │
   │   (能量送完 → 切 empty)                        │
   └──────────────────────────────────────────────┘
   │   (还有能量 → 继续 DELIVERING 找下一目标)
   ▼
```

### 取货优先级

```
1. 地面掉落资源 (amount >= 50)        ← 加锁
2. Tombstone (store >= 20)            ← 加锁
3. Container (store >= 100)
   ├─ store >= 500                    ← 不加锁（支持多搬运者）
   └─ store < 500                     ← 加锁
4. Storage (store >= 500)             ← 加锁
```

### 送货优先级

```
extension(1) > spawn(2) > tower(3) > storage(4) > container(5)

特殊情况：
- Spawn 能量 < 200   → 优先级 0（最高）
- Tower 能量 < 500   → 优先级 2.5
- 同优先级按剩余容量降序（最空的优先）
- 每个目标最多 3 个搬运者同时锁定
- 全满/全锁 → 兜底 Storage
```

## 10.4 让位协调流程

```
collector 连续 3 次无法到达 source (ERR_NO_PATH)
  │
  ├─ taskScheduler.requestYield(creep, source)
  │   ├─ 找 source 1 格内 FIND_CREEPS
  │   ├─ 优先驱离非采集者 (transporter/upgrader/builder/repairer)
  │   │   └─ _findYieldPosition(source) 找让位目标
  │   │       └─ 设置 c.memory._yieldUntil = Game.time + 5
  │   │           c.memory._yieldTarget = {x, y, roomName}
  │   │           打印日志 → return true
  │   └─ 其次调整其他 collector/harvester 站位（同上）
  │
  ├─ 让位成功 → 清空失败计数，下一 tick 重新尝试
  └─ 无让位者 → assignedSourceId=null，标记冷却 100 ticks，换 source

被请求者下一 tick run() 入口:
  └─ taskScheduler.checkYield(creep)
      ├─ 无 _yieldUntil → return false（正常执行）
      ├─ Game.time > _yieldUntil → 清理标记 → return false
      └─ 仍在让位期
          └─ moveTo(_yieldTarget, {stroke:'#ff4444'}) → return true（跳过本 tick）
```

### 让位目标查找（`_findYieldPosition`）

```
从 r = YIELD_DISTANCE(3) 到 YIELD_DISTANCE+2(5) 逐层外圈查找:
  ├─ room.lookAtArea(minY, minX, maxY, maxX, true) 一次性获取区域
  ├─ 仅检查 Chebyshev 距离 = r 的外圈格
  ├─ 跳过 terrain=wall
  ├─ 跳过非 road/rampart 的结构
  ├─ lookForAt(LOOK_CREEPS) 检查无 creep
  └─ 返回首个可站立格 {x, y, roomName}
```

## 10.5 Collector 采集与投放流程

```
run(creep)
  │
  ├─ checkYield(creep) → 让位中则 return
  ├─ _healthCheck(creep) → 卡住则进入恢复模式 return
  │
  ├─ 能量空 → collectorState = 'harvesting'
  ├─ 能量满 → collectorState = 'dropping'
  │
  ├─ harvesting:
  │   └─ _doHarvest
  │       ├─ _getAssignedSource(creep)
  │       │   ├─ 已有绑定且未满载 → 保持
  │       │   ├─ 满载/冷却 → 换近矿（按 spawn 距离）
  │       │   └─ 全满 → 兜底最近未冷却矿
  │       ├─ 未到达 → moveTo(source, {reusePath:10})
  │       │   └─ ERR_NO_PATH → 失败计数 → requestYield/换矿
  │       └─ 到达 → harvest(source)
  │           ├─ OK → _updateHealth，接近满仓切 dropping
  │           ├─ ERR_NOT_ENOUGH_RESOURCES → _updateHealth（再生中）
  │           ├─ ERR_NOT_OWNER → 换 source
  │           └─ 其他 → _handleHarvestError（指数退避）
  │
  └─ dropping:
      └─ _doDrop
          ├─ _findNearestContainer(creep)  范围内 (<=3) 最近可用 Container
          │   ├─ 有 → transfer
          │   │   ├─ OK → 继续投放或切 harvesting
          │   │   ├─ ERR_NOT_IN_RANGE → moveTo
          │   │   └─ ERR_FULL → 标记满，找下一个；全满则 drop
          │   └─ 无 → drop(RESOURCE_ENERGY)  drop mining
```

## 10.6 房间变更恢复流程

```
检测到 spawn.room.name 与 state.spawnRoomName 不同
  │
  ├─ _purgeRoomState()
  │   ├─ 清理 Memory.tasks.active 中 source/target 失效的任务
  │   ├─ 清理 Memory.tasks.targetLocks 失效锁
  │   ├─ 清理 Memory._transporterLocks.pickups/delives 失效锁
  │   └─ 清理 Memory._transporterLockTimestamps
  │
  ├─ state._cacheReady = false  触发下 tick 重新初始化
  ├─ state.sourceIds = []
  ├─ state.sourceData = {}
  ├─ state.sourceSpawnDist = {}
  ├─ state.towerIds = []  触发 Tower 重新扫描
  └─ 打印 [Main] 检测到房间变更: 旧 → 新，缓存已重置
```
