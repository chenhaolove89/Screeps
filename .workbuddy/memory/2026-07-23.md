# 2026-07-23 — Screeps 项目架构重构

## 完成工作

### 1. 新增统一任务调度框架 (`task.scheduler.js`)
- 任务 CRUD：创建、分配、状态更新、完成/失败
- 优先级队列：CRITICAL(0) → HIGH(1) → NORMAL(2) → LOW(3)
- 资源锁机制：sourceLocks + targetLocks 防止多 creep 竞争
- 死锁检测：检查死亡 creep 残留锁、超时停滞任务
- 任务去重：相同 source→target 的活跃任务不重复创建
- 自动超时回收：TASK_TIMEOUT=300 ticks
- 统计上报：active/completed/failed/avgDuration
- 所有状态持久化在 `Memory.tasks` 中

### 2. 新增搬运者角色 (`role.transporter.js`)
- 完整状态机：IDLE → GET_TASK → MOVING_TO_PICKUP → PICKING_UP → MOVING_TO_DELIVER → DELIVERING
- 取货优先级：掉落资源 > Tombstone/Ruin > Container(≥100) > Storage(≥500)
- 送货优先级：Spawn低能量(0) > Extension(2) > Tower低能量(2.5) > Tower(3) > Storage
- 资源锁：独立于调度器的轻量级取货/送货锁
- 错误处理：移动/取货/送货各 3-5 次失败后放弃目标
- 停滞检测：50 ticks 无移动自动重置

### 3. 优化采集者角色 (`role.collector.js`)
- 职责分离：纯采集，不搬运（原 harvester 同时做两件事）
- 智能源分配：自动分配到采集者最少的 source（max 2 per source）
- 就近投放：优先 transfer 到附近 Container
- 指数退避重试：`baseInterval * 2^(retry-1)`，max 100 ticks
- 健康检查：50 ticks 无移��/采集 → 进入恢复模式（20 ticks 冷却）
- Room.find 缓存：20 ticks TTL，避免每 tick 全量扫描
- 结构化日志：DEBUG/INFO/WARN/ERROR 四级

### 4. 更新集成模块
- `config.js`：添加 collector(3)/transporter(2) 角色数量；harvester 设为 0；新增 scheduler/collector/transporter 配置节
- `state.js`：新增 schedulerStats 状态字段
- `manager.spawn.js`：
  - 孵化队列新增 collector(最高优先级) + transporter
  - Collector body: 2WORK+1MOVE per unit (250 energy)，专注采集
  - Transporter body: 1WORK+2CARRY+2MOVE per unit (300 energy)，专注搬运
- `main.js`：
  - 集成 taskScheduler（init + checkTimeouts + checkDeadlocks）
  - 死亡 creep 清理时释放调度器锁
  - 角色调度新增 collector 和 transporter
  - 每 100 tick 输出调度统计

### 架构变更要点
- **原 harvester 保持不变**（roleTargets.harvester=0 禁用），向后兼容
- **Collector ↔ Transporter 协作**：Collector 采完投放 Container，Transporter 从 Container/地面取货运送
- **调度器 vs Transporter 锁**：调度器管理正式任务锁，Transporter 的轻量锁用于防止多个 transporter 同时冲向同一个临时目标

## Bug 修复（同日）

### Bug 1: Collector body 不含 CARRY → 无法存储能量
**症状**：所有 Collector 孵化后 body 不含 CARRY 部件（`2×WORK + 1×MOVE`），`creep.store` 容量为 0，采集的能量无处存放。
**修复** (`manager.spawn.js`): Collector body 公式改为 `2×WORK + 1×CARRY + 1×MOVE = 300/unit`

### Bug 2: harvest ERR_NOT_IN_RANGE 被误判为未知错误
**症状**：`creep.harvest()` 在未到达 source 时返回 `-4`（当前环境映射），代码中的 `result === ERR_NOT_IN_RANGE`（值 `-9`）无法匹配，落入 else 分支触发指数退避。
**修复** (`role.collector.js`): 用 `creep.pos.inRangeTo(source, 1)` 距离检查代替硬编码错误码判断。只有真正的不可恢复错误（ERR_NO_PATH 路径不可达）才退避。

### Bug 3: Transporter moveTo 过于严格
**症状**：`creep.moveTo()` 返回值 `-4`（非标准错误码）被判定为移动失败，触发重试计数，5 次后放弃目标。
**修复** (`role.transporter.js`): 只对 `ERR_NO_PATH` 和 `ERR_INVALID_TARGET` 累计错误计数，其余返回值不影响重试逻辑。

### Bug 4: Transporter 状态机每 tick 被重置（死锁）
**症状**：`run()` 中 `creep.store[RESOURCE_ENERGY]===0` 每 tick 都将 `transporterState` 重置为 `GET_TASK`，导致 `MOVING_TO_PICKUP` 刚设置就被清除。moveTo 执行了但下一 tick 又重新找目标，永远无法移动到取货点。
**修复** (`role.transporter.js`): 引入 `_transportPhase`（'empty'/'carry'）追踪高层次阶段，只在阶段切换时重置状态。状态机按 phase 分两路处理。

### Bug 5: Collector 无 Container 时永不释放能量
**症状**：`_doDrop` 在无 Container 时只切换回 `harvesting` 状态，不丢能量。Collector 满仓后既不能继续采也不能投递，形成死锁。Transporter 也因此无货可搬。
**修复** (`role.collector.js`): 无 Container 时调用 `creep.drop(RESOURCE_ENERGY)` 将能量丢在地面，形成 drop mining 模式。

### Bug 6: Transporter 缓存 TTL 过长
**症状**：Room.find 缓存 15 ticks 刷新一次，Collector 刚丢下的能量不能被及时发现。
**修复** (`role.transporter.js`): CACHE_TTL 从 15 缩短为 5 ticks。
